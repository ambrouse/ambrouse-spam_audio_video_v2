# Release Notes - v0.1.6

## Highlights

- Main video pipeline now has one production path: Rust/D3D11/NVENC full
  timeline render, then optional audio mux.
- Removed the old segmented clip render/combine fallback from runtime source.
- Audio combine now streams WAV blocks directly to `combined.wav`; RAM usage no
  longer scales with full story duration during merge.
- Production defaults target `1080p60`, hardware H.264, and `VIDEO_PRESET=throughput`.
- CI now validates Python policy tests plus the Windows Rust renderer build.
- Windows setup/backend now prefer the short VieNeu runtime path such as
  `D:\.vieneu-<hash>` before the compatibility junction. This avoids Python
  package scans crossing the 260-character path limit during TTS prewarm in
  extracted portable releases.

## Long-Run Stability

This release is aimed at 20-40 hour production runs:

- no per-clip FFmpeg render process accumulation in the main video stage;
- no generated clip folder contract in the main pipeline;
- story renderer loads unique scene textures once per render and encodes one
  continuous H.264 stream;
- TTS worker keeps one warm model process and writes final audio through a
  streaming merge;
- unit tests block reintroducing `clips_native`, `SPAM_VIDEO_NATIVE`, and
  non-streaming audio concatenate code.

## Runtime Contract

- Audio model remains `JayLL13/VoxCPM-1.5-VN`.
- VoxCPM2 experiment artifacts were removed; that model was slower and lower
  quality in local review.
- Video renderer source lives at `spam_audio_video/renderers/story_gpu_renderer`.
- Optional renderer binary override: `SPAM_VIDEO_RENDERER_BIN`.
- On Windows, TTS worker launch prefers `SPAM_TTS_PYTHON` or the short
  `.vieneu-<hash>` runtime generated by setup, then falls back to compatibility
  paths.
- The main video path requires 16:9 `60fps` `h264_nvenc`; unsupported settings
  fail early instead of falling back to old code.
